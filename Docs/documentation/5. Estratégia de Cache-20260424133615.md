# 5. Estratégia de Cache

### 5.1 Por que cachear
Cada chamada ao Omie tem latência média de 300–800ms e existe risco de rate limiting. Em uma sessão de revisão típica, o analista navega por múltiplas páginas e executa ações que disparam novas buscas dos mesmos dados. Sem cache, cada troca de página na tela de revisão geraria dezenas de chamadas repetidas ao Omie para os mesmos lançamentos.
### 5.2 Nível 1 — Cache de contas correntes (banco de dados)
**O que cacheia:** lista de contas correntes (`nCodCC`, nome, banco, tipo) de cada cliente.
**Onde fica:** tabela `omie_accounts_cache` no PostgreSQL.
**TTL:** 24 horas a partir de `synced_at`.
**Fluxo de leitura:**
1. Backend recebe request que precisa das contas do cliente
2. Consulta `omie_accounts_cache` onde `client_id = X` e `synced_at > now() - 24h`
3. Se encontrar registros válidos → retorna do cache (não chama Omie)
4. Se não encontrar ou expirado → chama `ListarContasCorrentes` no Omie, atualiza cache e retorna
**Invalidação manual:**
*   Botão "Sincronizar contas" na tela de detalhe do cliente
*   Chama `POST /api/clients/{id}/sync-accounts`
*   Força chamada ao Omie e atualiza cache independente do TTL
**Quando o cache expira naturalmente:**
*   Após 24h, na próxima request que precisar das contas, o cache é atualizado automaticamente
### 5.3 Nível 2 — Cache de lançamentos Omie (memória)
**O que cacheia:** dados de lançamentos individuais do Omie (fornecedor, categoria, valor, status) identificados por `omie_lancamento_id`. Usados na tela de revisão para enriquecer as linhas conciliadas e as divergências.
**TTL:** 2 horas — suficiente para cobrir uma sessão de revisão sem risco de dado muito desatualizado.
**Chave do cache:** `{client_id}:{omie_lancamento_id}`
**Opção A — Map em memória:**
*   Estrutura: dicionário/Map onde a chave é a string acima e o valor é `{ data: {...}, expiresAt: timestamp }`
*   Implementar como singleton na camada de serviço
*   Limpeza: verificar `expiresAt` antes de cada leitura; implementar limpeza periódica (ex: a cada hora) para evitar memory leak
*   Limitação: perdido ao reiniciar o servidor; não compartilhado entre instâncias do processo Python (problema em ambientes com múltiplas instâncias)
**Opção B — Redis:**
*   Chave: `omie_lancamento:{client_id}:{omie_lancamento_id}`
*   TTL configurado com `SETEX` (define expiração automática)
*   Compartilhado entre instâncias; persiste entre deploys
*   Requer serviço Redis na infra (Redis Cloud, Upstash, etc.)
*   Abstração recomendada: criar interface de cache com métodos `get(key)` e `set(key, value, ttlSeconds)` que pode ser implementada tanto com Map quanto com Redis — facilita trocar a implementação sem refatorar os endpoints que consomem o cache
**Fluxo de leitura (igual para as duas opções):**
1. Backend recebe lista de `omie_lancamento_id` para enriquecer
2. Verifica quais IDs já estão no cache (e não expiraram)
3. Para os IDs não encontrados ou expirados: chama Omie em batch ou individualmente
4. Popula o cache com os novos dados
5. Retorna os dados de todos os IDs (combinando cache + Omie)

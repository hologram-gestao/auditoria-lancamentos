# 4. Segurança e Criptografia

### 4.1 Criptografia AES-256-GCM
O algoritmo AES-256-GCM foi escolhido porque, além de criptografar o dado, gera uma tag de autenticação que garante integridade — qualquer alteração no dado criptografado é detectada na descriptografia.
**Fluxo de criptografia ao salvar:**
1. Gerar IV (Initialization Vector) aleatório de 12 bytes para cada operação de criptografia
2. Criptografar o dado usando AES-256-GCM com a chave de ambiente e o IV gerado
3. Salvar no banco: dado criptografado (em hex ou base64), IV e a tag GCM
4. Nunca reusar o mesmo IV para campos diferentes ou registros diferentes
**Fluxo de descriptografia ao ler:**
1. Buscar do banco: dado criptografado, IV e tag GCM
2. Usar a chave de ambiente para descriptografar
3. A tag valida automaticamente a integridade — se houver adulteração, a operação falha
4. Retornar o dado em texto plano apenas em memória, nunca persistir de volta em aberto
**Onde fica a chave:**
*   Variável de ambiente `OMIE_ENCRYPTION_KEY` no servidor Python (configurada no ambiente de deploy)
*   Nunca no banco de dados
*   Nunca em logs de aplicação
*   Nunca em respostas de API
*   Rotação da chave requer re-criptografar todos os registros afetados
### 4.2 Credenciais Omie — fluxo completo
As credenciais Omie dos clientes percorrem o seguinte caminho em toda requisição ao Omie:

```markdown
1. Backend recebe request autenticado (JWT válido)
2. Backend busca do banco: omie_app_key_encrypted, omie_app_secret_encrypted, encryption_iv, encryption_tag
3. Backend descriptografa em memória usando OMIE_ENCRYPTION_KEY
4. Backend usa as credenciais na chamada POST ao Omie
5. Omie responde
6. Backend descarta as credenciais em texto plano (garbage collected)
7. Backend retorna apenas os dados relevantes ao frontend
```

**Garantias críticas:**
*   Credenciais nunca aparecem em logs (usar `[REDACTED]` se necessário logar a chamada)
*   Credenciais nunca são retornadas em respostas de API — nem ao frontend, nem em erros
*   Se o banco for comprometido, as credenciais permanecem inacessíveis sem `OMIE_ENCRYPTION_KEY`
### 4.3 Autenticação JWT
*   **Access token:** JWT assinado com chave de ambiente, expiração de 1 hora. Contém `user_id` e `role`.
*   **Refresh token:** JWT com expiração de 7 dias. Usado para obter novos access tokens sem relogin.
*   **Armazenamento no cliente:** cookies `HttpOnly` e `Secure` (não acessíveis via JavaScript). Alternativa: `localStorage` apenas se o time decidir, com ciência dos riscos de XSS.
*   **Invalidação:** ao desativar um usuário (`active = false`), o middleware valida `active` no banco a cada request — usuário perde acesso mesmo com token ainda válido.
### 4.4 Controle de acesso por role
O middleware injeta [`user.id`](http://user.id/) e `user.role` em toda requisição. Cada endpoint verifica:
**Admin:**
*   Acessa qualquer cliente sem verificar `client_assignments`
*   Acessa qualquer sessão de conciliação
*   Acessa rotas `/configuracoes/*`
**Manager:**
*   Acessa apenas clientes onde `client_assignments.user_id =` [`user.id`](http://user.id/)
*   Qualquer tentativa de acessar cliente de outro gerente retorna HTTP 403
*   Não acessa rotas `/configuracoes/*`

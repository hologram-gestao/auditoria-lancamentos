### Tabela: `omie_accounts_cache`
Cache das contas correntes por cliente. Evita chamadas repetidas ao Omie para dados que raramente mudam. TTL de 24h.

| Coluna | Tipo | Restrições | Descrição |
| ---| ---| ---| --- |
| `id` | UUID | PK, NOT NULL | — |
| `client_id` | UUID | FK → [clients.id](http://clients.id), NOT NULL | — |
| `omie_conta_id` | BIGINT | NOT NULL | `nCodCC` do Omie |
| `name` | VARCHAR(200) | NOT NULL | Nome da conta (`descricao`) |
| `bank_name` | VARCHAR(100) | NOT NULL | Código do banco vindo do Omie (`codigo_banco`, ex: "748"). O endpoint Omie não devolve o nome por extenso — mapeamento código→nome é responsabilidade do front. |
| `account_type` | VARCHAR(10) | NOT NULL | `CC` (Conta Corrente), `CA` (Cartão), `CX` (Caixinha) ou outros — front tolera tipos desconhecidos |
| `synced_at` | TIMESTAMPTZ | NOT NULL | Momento da última sincronização |

**Constraint:** `UNIQUE (client_id, omie_conta_id)`
**Regra de uso:**
*   Se `synced_at < now() - interval '24 hours'` → busca do Omie, atualiza o cache e retorna
*   Caso contrário → retorna do cache diretamente
*   Botão "Sincronizar" manual disponível na tela do cliente para forçar atualização no mesmo dia

## ESTRATÉGIA DE CACHE
Dois níveis de cache para minimizar chamadas ao Omie sem perder consistência dos dados.
### Nível 1 — Cache em banco (`omie_accounts_cache`)
Para contas correntes por cliente. Dados que mudam raramente — uma nova conta bancária é adicionada esporadicamente.
*   TTL: 24 horas
*   Invalidação: automática por TTL ou manual via botão "Sincronizar" na tela do cliente
*   Implementação: verificar `synced_at` antes de cada uso — se expirado, buscar do Omie e atualizar
### Nível 2 — Cache em memória para lançamentos Omie durante revisão
Para dados de lançamentos buscados durante a tela de revisão (`omie_lancamento_id → fornecedor, categoria, valor`). O analista navega por páginas repetidamente — sem cache, cada troca de página gera chamadas ao Omie para os mesmos IDs.
**Opção A — Map em memória (MVP)**
*   Implementar como `Map<string, { data, expiresAt }>` na camada de serviço
*   Chave: `{client_id}:{omie_lancamento_id}`
*   TTL: 2 horas (suficiente para uma sessão de revisão)
*   Sem dependência de infraestrutura adicional
*   Limitação: cache perdido ao reiniciar o servidor; não compartilhado entre instâncias
**Opção B — Redis**
*   Chave: `omie_lancamento:{client_id}:{omie_lancamento_id}`
*   TTL: 2 horas com `SETEX`
*   Cache persiste entre deploys e é compartilhado entre instâncias
*   Requer serviço Redis na infraestrutura (Redis Cloud, Upstash, etc.)
**Recomendação:** validar com o time. Para o MVP com instância única, o Map resolve sem overhead de infra. Se o produto escalar para múltiplas instâncias, migrar para Redis — a interface do serviço de cache pode ser abstraída para facilitar essa troca sem refatorar o restante do código.

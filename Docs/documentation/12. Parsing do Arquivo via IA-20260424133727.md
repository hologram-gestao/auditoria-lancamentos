# 12. Parsing do Arquivo via IA

### 12.1 O que acontece após a verificação de duplicata
Após confirmar que não é duplicata, o frontend envia o arquivo ao backend para extração via Claude API.
**Enquanto processa:**
*   Botão "Processar" vira "Processando..." com spinner
*   Todos os campos do formulário ficam desabilitados
*   Exibir progress indicator com mensagem: "Extraindo movimentações com IA... Isso pode levar até 60 segundos."
### 12.2 O que o backend faz
1. Recebe o arquivo via multipart/form-data
2. Converte o arquivo para base64 em memória no servidor
3. Envia para a Claude API com prompt de extração estruturada
4. O arquivo **não é salvo em disco ou storage** — é processado apenas em memória
5. Aguarda a resposta (timeout: 60 segundos)
6. Parseia a resposta JSON da Claude API
7. Retorna ao frontend a lista de transações e os metadados
**O que a Claude API recebe:**
*   O arquivo em base64 como `document` block (para PDF) ou como dado bruto
*   Um prompt de sistema instruindo a extrair transações no formato JSON estruturado
*   O prompt especifica: extrair todas as movimentações (sem excluir nenhuma linha), padronizar datas para ISO 8601, valores com sinal (positivo = crédito, negativo = débito), preservar descrição exata
**O que a Claude API retorna (JSON):**
*   `bank_name`: nome do banco identificado no documento
*   `account_type`: `"checking"` (conta corrente) ou `"credit_card"` (cartão)
*   `period_start` e `period_end`: datas no formato YYYY-MM-DD
*   `opening_balance` e `closing_balance`: saldos inicial e final
*   `transactions[]`: lista de objetos com `date`, `description`, `amount`, `balance`
**Tratamento de erros do parsing:**
*   Timeout (>60s): retornar erro "O processamento demorou mais que o esperado. Tente novamente."
*   Claude API retorna erro 5xx: 1 retry automático. Se persistir: "Serviço de extração temporariamente indisponível."
*   JSON inválido ou `transactions` vazio: "Não foi possível extrair movimentações do arquivo. Verifique se o arquivo está íntegro e sem proteção por senha."
### 12.3 Tela de validação do parsing (prévia)
**Como aparece:** o formulário é substituído pela tela de prévia (na mesma página — não navega para outra rota).
**Elementos visíveis:**
*   Título: "Confirme as movimentações extraídas"
*   Card de metadados:
    *   "Banco identificado:" {banco}
    *   "Tipo de conta:" Conta Corrente ou Cartão de Crédito
    *   "Período:" {data\_inicio} a {data\_fim}
    *   "Saldo inicial:" R$ {valor formatado em BRL}
    *   "Saldo final:" R$ {valor formatado em BRL}
    *   "Total de transações:" {N} movimentações encontradas
*   Tabela de prévia com as **primeiras 5 transações**:
    *   Colunas: Data · Descrição · Valor
    *   Valores positivos (crédito): texto verde
    *   Valores negativos (débito): texto vermelho
*   Texto: "Verifique se os dados acima correspondem ao arquivo enviado antes de confirmar."
*   Dois botões:
    *   "Cancelar" (secundário): descarta todos os dados extraídos e volta ao formulário. Nenhum dado foi salvo até este ponto.
    *   "Confirmar e processar" (primário): inicia o processamento (navegação para tela de progresso)
**O que acontece ao confirmar:**
1. Frontend envia ao backend: lista de transações + metadados + `omie_conta_id` + `reference_month` + `date_tolerance_days` + `file_hash` + banco identificado
2. Backend cria a sessão e retorna `session_id`
3. Frontend navega para `/clientes/{client_id}/conciliacao/{session_id}/processando`

# 6. Integração com API do Omie

### 6.1 Autenticação no Omie
O Omie usa autenticação por par de chaves enviadas em cada requisição no body. Não há sessão ou token — toda chamada inclui `app_key` e `app_secret`.
**Estrutura base de toda requisição:**

```css
POST https://app.omie.com.br/api/v1/{módulo}/{endpoint}/
Content-Type: application/json

Body:
{
  "call": "{nome_do_método}",
  "app_key": "{app_key_do_cliente_descriptografada}",
  "app_secret": "{app_secret_do_cliente_descriptografada}",
  "param": [ { ...parâmetros } ]
}
```

**Estrutura base de resposta de sucesso:**
*   HTTP 200 com body contendo os dados solicitados
*   O campo de dados varia por endpoint (ex: `ListarContasCorrentes`, `extrato`, `cadastro`). O Omie é inconsistente — alguns endpoints reusam o nome do método como chave (PascalCase), outros usam snake_case. **Sempre validar contra resposta real**, não contra documentação interna.
**Estrutura de resposta de erro:**
*   HTTP 200 (mesmo em caso de erro do Omie — isso é uma particularidade da API)
*   Body contém `faultstring` (mensagem do erro) e `faultcode` (código numérico)
*   Todo handler de resposta Omie **deve verificar a presença de** **`faultstring`** **antes de processar os dados**
### 6.2 Endpoints utilizados neste módulo
#### `ListarContasCorrentes` — Listar contas correntes do cliente
**Endpoint:** `POST` [https://app.omie.com.br/api/v1/geral/contacorrente/](https://app.omie.com.br/api/v1/geral/contacorrente/)
**Quando usar:** ao abrir a tela de detalhe do cliente ou o formulário de nova conciliação, para popular o select de contas bancárias. Sempre verificar cache antes (Nível 1).
**Parâmetros enviados:**
*   `pagina`: número da página (começa em 1)
*   `registros_por_pagina`: 100 (máximo recomendado)
*   `apenas_importado_api`: `"N"` (retorna todas as contas, não apenas as criadas via API)
**Campos relevantes da resposta** (validados contra a doc oficial e response real em 19/05/2026):
*   Envelope: `pagina`, `total_de_paginas`, `registros`, `total_de_registros`, `ListarContasCorrentes` (array — sim, mesmo nome do método em PascalCase).
*   Itens em `ListarContasCorrentes[]`:
    *   `nCodCC` (int): ID único da conta no Omie — **usar como referência em todas as operações**.
    *   `descricao` (string): nome da conta (ex: "Sicredi 91263-1"). Único campo "humano" disponível aqui.
    *   `codigo_banco` (string, 3 dígitos): código do banco (ex: "748" Sicredi, "341" Itaú, "237" Bradesco). O Omie **não devolve o nome do banco por extenso** neste endpoint — se precisar, mapear do código localmente.
    *   `tipo_conta_corrente` (string, 2 chars): código do tipo de conta. Doc oficial declara 13 valores — `AC` Administradora de Cartões, `AD` Adiantamento, **`CA` Conta Aplicação**, `CC` Conta Corrente, `CE` Conta Empréstimo, `CG` Conta Garantida, `CN` Crediário/Carnê, `CP` Conta Poupança, **`CR` Cartão de Crédito**, `CV` Carteira Virtual, `CX` Caixinha, `MT` Mútuo, `PG` Conta de Pagamento. **Aceitar todos**: o front lida com tipos desconhecidos exibindo o valor cru. **Atenção (auditoria M-1):** `CA` **não é** cartão — cartão é `CR`.
*   Outros campos disponíveis (não consumimos hoje): `cCodCCInt`, `codigo_agencia`, `numero_conta_corrente`, `saldo_inicial`, `valor_limite`, `inativo`, `bloqueado`.
**Paginação:** se `ListarContasCorrentes` retornar exatamente 100 itens, buscar página 2 e assim por diante até retornar menos de 100.

> **Atenção (lição aprendida — 19/05/2026):** a v1 desta doc descrevia os campos como `lista_conta_corrente`, `nCodBanco`, `descricaoBanco`, `tipo` — todos errados. Schema, mock e testes foram escritos contra a doc errada e ficaram internamente consistentes, com CI verde, mas em produção o Omie devolvia sempre lista vazia. **Para qualquer novo endpoint Omie, validar o response real antes de descrever na doc.**
#### `ListarClientes` — Verificar conexão com Omie
**Endpoint:** `POST` [https://app.omie.com.br/api/v1/geral/clientes/](https://app.omie.com.br/api/v1/geral/clientes/)
**Quando usar:** exclusivamente para o "Testar conexão" no cadastro de cliente. Faz a requisição mais simples possível para validar as credenciais antes de salvar.
**Parâmetros enviados:**
*   `pagina`: 1
*   `registros_por_pagina`: 1
**O que importa na resposta:** apenas saber se não veio `faultstring`. Se a resposta for bem-sucedida (HTTP 200 sem `faultstring`), as credenciais são válidas.
#### `ListarExtrato` — Lançamentos realizados na conta
**Endpoint:** `POST` [https://app.omie.com.br/api/v1/financas/extrato/](https://app.omie.com.br/api/v1/financas/extrato/)
**Quando usar:** durante o processamento automático (Background 8.2) para buscar os lançamentos que serão usados no cruzamento com o arquivo.
**Parâmetros enviados:**
*   `nCodCC`: ID da conta corrente (`omie_conta_id` da sessão)
*   `cCodIntCC`: `""` (vazio — usar apenas quando a referência for o código de integração)
*   `dPeriodoInicial`: data de início no formato `DD/MM/YYYY` — usar `period_start` do arquivo menos `date_tolerance_days` dias
*   `dPeriodoFinal`: data de fim no formato `DD/MM/YYYY` — usar `period_end` do arquivo mais `date_tolerance_days` dias
> **Importante:** ampliar o período em `date_tolerance_days` para cada lado para capturar lançamentos que estejam nas bordas do período — por exemplo, uma movimentação do dia 31 no banco pode estar lançada no Omie como dia 2 do mês seguinte.
**Envelope da resposta** (`eccListarExtratoResponse`, validado em 20/05/2026 contra doc oficial):
*   `listaMovimentos[]` ← chave canônica da lista (**não** `extrato`, como a v1 dizia).
*   Não há envelope de paginação. Doc oficial não documenta limite máximo; assumimos response inteiro em uma chamada (ver Auditoria ALTO-3).
**Campos relevantes de cada item em `listaMovimentos[]`** (alias case-sensitive da Omie ↔ atributo Python):
*   `nCodLancamento` (int) → `n_cod_lancamento`. ID único do lançamento — armazenar em `omie_lancamento_id`.
*   `nCodLancRelac` (int, opcional) → `n_cod_lanc_relac`. Lançamento relacionado (parcelamento) — não usado no matching atual, persiste no cache para uso futuro.
*   `cNatureza` (string1) → `c_natureza`. `"D"` (Débito = saída) ou `"C"` (Crédito = entrada).
*   `dDataLancamento` (string `DD/MM/AAAA`) → `d_data_lancamento`.
*   `nValorDocumento` (decimal) → `n_valor_documento`. Valor absoluto (sempre positivo).
*   `cSituacao` (string40) → `c_situacao`. Status: valores observados em prática — `Conciliado`, `Atrasado`, `Previsto`. Doc oficial NÃO enumera; código tolera valores fora do enum.
*   `cObservacoes` (text) → property `description`. Texto livre — único campo "humano" estável.
*   `cRazCliente` (string60) → property `supplier` (preferida). Razão social do cliente/fornecedor.
*   `cDesCliente` (string100) → property `supplier` (fallback). Nome fantasia.
*   `cCodCategoria` (string20) → property `category` (fallback). Código curto (ex: "DT").
*   `cDesCategoria` (string350) → property `category` (preferida). Descrição completa.
**Normalização obrigatória:**
*   `cNatureza = 'D'` → aplicar sinal negativo ao valor (`nValorDocumento * -1`). Implementado em `LancamentoExtrato.signed_amount`.
*   `cNatureza = 'C'` → manter positivo.
*   `dDataLancamento`: converter de `DD/MM/AAAA` para objeto `date` Python (validator no schema).
**Outros campos disponíveis** (não consumidos hoje): `nSaldo`, `nSaldoPrev`, `cTipoDocumento`, `cNumero`, `cDocumentoFiscal`, `cParcela`, `dDataConciliacao`, `cBloqueado`.
#### `ListarContasPagar` — Títulos a pagar previstos e atrasados
**Endpoint:** `POST` [https://app.omie.com.br/api/v1/financas/contapagar/](https://app.omie.com.br/api/v1/financas/contapagar/)
**Quando usar:** durante o processamento automático (Background 8.3) para detectar lançamentos Omie que **não** aparecem no arquivo bancário (situação `omie_sem_arquivo`).
**Parâmetros enviados** (validados contra doc oficial + chamada real em 19/05/2026):
*   `pagina` (int): número da página
*   `registros_por_pagina` (int): 50
*   `filtrar_por_data_de` (string `DD/MM/YYYY`): primeiro dia do mês de referência
*   `filtrar_por_data_ate` (string `DD/MM/YYYY`): último dia do mês de referência
*   `filtrar_conta_corrente` (int — **sem `por_`**, doc oficial `lcpListarRequest`): `omie_conta_id` da sessão. Cuidado: a v1 desta doc dizia `filtrar_por_conta_corrente` — em prod o Omie devolve **HTTP 500 + `OmieAPI-Error: 5001 - Tag [FILTRAR_POR_CONTA_CORRENTE] não faz parte da estrutura do tipo complexo [lcpListarRequest]!`**.
*   `filtrar_por_status` (string, max 100 chars): valores aceitos pela doc oficial — `CANCELADO`, `PAGO`, `LIQUIDADO`, `EMABERTO`, `PAGTO_PARCIAL`, `VENCEHOJE`, `AVENCER`, `ATRASADO`. **Não usar `"PREVISTO"`** (não está no enum, Omie devolve 5xx). Para o matching atual usamos `ATRASADO` e `AVENCER` (= "previsto" na nomenclatura interna do projeto). A doc oficial cita aceitar CSV (`"AVENCER,ATRASADO"`); hoje fazemos chamadas separadas para preservar granularidade do log.
**Envelope da resposta** (`lcpListarResponse`):
*   `pagina`, `total_de_paginas`, `registros`, `total_de_registros`
*   `conta_pagar_cadastro[]` ← chave da lista (**não** `cadastro`, como a v1 da doc dizia)
**Campos relevantes de cada item em `conta_pagar_cadastro[]`** (validados contra doc oficial em 20/05/2026):
*   `codigo_lancamento_omie` (int): ID do lançamento — **único dado armazenado localmente** como `omie_lancamento_id`.
*   `data_vencimento` (string `DD/MM/AAAA`): data do vencimento.
*   `valor_documento` (decimal): valor do título.
*   `codigo_cliente_fornecedor` (int): ID do cliente/fornecedor no Omie. **Não há nome legível na response deste endpoint** — resolver via `ListarClientes` (campo `codigo_cliente_omie`) em batch quando a feature de nome legível for implementada.
*   `codigo_categoria` (string20): código da categoria (ex: "DT"). **Sem descrição neste endpoint** — resolver via futuro consumer de `ListarCategorias`.
*   `numero_documento` (string20) / `numero_documento_fiscal` (string20): úteis pra rastreio.
*   `observacao` (text): texto livre — útil pra exibir contexto na revisão.
*   `status_titulo` (`string3`): valores não enumerados na doc oficial. Tratado como string livre. **Não confundir com o filtro `filtrar_por_status`** (que aceita palavras inteiras como `ATRASADO`/`AVENCER`); auditoria A-2 pendente para confirmar formato real via fixture.

> **Lição aprendida (20/05/2026):** a v1 desta doc declarava `nome_fornecedor` e `descricao_categoria` que NÃO existem no response oficial — Pydantic deixava-os `None` silenciosamente. Caso documentado em `Docs/AUDITORIA_OMIE_INTEGRACAO.md` CRÍTICO-5. Hoje o schema reflete o response real e expõe só os códigos.
#### `ListarContasReceber` — Títulos a receber previstos e atrasados
**Endpoint:** `POST` [https://app.omie.com.br/api/v1/financas/contareceber/](https://app.omie.com.br/api/v1/financas/contareceber/)
**Quando usar:** mesmo contexto que `ListarContasPagar` — buscar recebíveis previstos/atrasados que não aparecem no arquivo.
**Estrutura:** mesmos parâmetros que `ListarContasPagar` (incluindo `filtrar_conta_corrente` sem `por_`). A doc oficial **não enumera** valores aceitos em `filtrar_por_status`, mas SDKs de terceiros usam o mesmo enum de ContasPagar com a troca de `PAGO/PAGTO_PARCIAL` por `RECEBIDO/RECEB_PARCIAL`. Chave da lista no response: **`conta_receber_cadastro[]`** (não `cadastro`).
### 6.3 Tratamento de erros Omie
Todo handler de resposta do Omie deve seguir esta sequência:
1. Verificar status HTTP. **5xx ambíguo:** se vier com header `OmieAPI-Error`, é erro **permanente** (ex: tag inválida, "Consumo redundante"); levantar `OmieFaultError` sem retry. Sem o header, é infra transitória — retry com backoff.
2. Parsear o body como JSON.
3. Verificar se existe `faultstring` no body → se sim, levantar erro com a mensagem do Omie.
4. Se não há `faultstring`, prosseguir com o processamento dos dados.
**Mensagens de erro mapeadas para o usuário:**
*   `faultstring` relacionado a credenciais inválidas → "Credenciais Omie inválidas. Verifique as configurações do cliente." (`OmieAuthError`).
*   Timeout HTTP sem resposta (≥ 15s) → "O Omie não respondeu no tempo esperado. Tente novamente." (`OmieTimeoutError`).
*   5xx persistente após retries (Omie respondeu, com instabilidade) → "O Omie está com instabilidade no momento. Tente novamente em instantes." (`OmieServerError`). Distinto de `OmieTimeoutError` — usar o nome certo para que o oncall não procure problema de rede quando o Omie respondeu com 500.
*   5xx + header `OmieAPI-Error: NNNN - ...` → propagar a mensagem do header em `OmieFaultError.user_message`.
*   Qualquer outro `faultstring` → exibir traduzido quando possível, ou "Ocorreu um erro ao acessar o Omie." (`OmieFaultError`).
**Timeout padrão:** 15 segundos para todas as chamadas ao Omie. **Timeout do parsing (Claude API):** 60 segundos — arquivos grandes demoram mais para ser processados.

> **Lição aprendida (19/05/2026):** validar nomes de parâmetros e chaves de envelope contra resposta real do Omie ANTES de descrever na doc interna. A v1 deste arquivo descrevia `filtrar_por_conta_corrente`, `lista_conta_corrente`, `nCodBanco`, `descricaoBanco`, `tipo`, `cadastro` — TODOS errados na API real. Schema/mocks/testes ficaram internamente consistentes e CI verde, mas em produção tudo quebrava com 5xx silencioso ou lista vazia.

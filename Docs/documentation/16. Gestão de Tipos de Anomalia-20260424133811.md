# 16. Gestão de Tipos de Anomalia

### 16.1 Acesso
Rota: `/configuracoes/anomalias`
Visível **apenas para Admin**. Menu de configurações aparece na sidebar apenas para Admin.
### 16.2 Tipos pré-cadastrados (seed)
O banco é populado com os seguintes tipos na migration inicial:

| Código | Nome | Severidade |
| ---| ---| --- |
| `missing_in_omie` | Movimentação sem lançamento no Omie | Crítica |
| `missing_in_file` | Lançamento Omie sem correspondente no extrato | Crítica |
| `wrong_account` | Lançamento possivelmente na conta errada | Crítica |
| `inconsistent_category` | Mesma descrição, categorias diferentes entre meses | Moderada |
| `category_mismatch_nature` | Categoria incompatível com a natureza do lançamento | Moderada |
| `internal_transfer_as_revenue` | Transferência interna classificada como receita | Crítica |
| `possible_duplicate` | Possível lançamento duplicado | Moderada |
| `classification_improvement` | Sugestão de padronização de categoria | Informativa |

### 16.3 Tela de gestão
**Elementos visíveis:**
*   Breadcrumb: "Configurações > Tipos de Anomalia"
*   Texto explicativo: "Os tipos de anomalia definem as categorias de problemas identificáveis durante a conciliação. Tipos inativos não aparecem para seleção manual pelos analistas."
*   Campo de busca por nome ou código
*   Tabela com colunas: Nome · Código · Severidade (badge) · Descrição · Status · Ações
*   Ordenação padrão: críticas primeiro, depois moderadas, depois informativas
**Ações por linha (Fase 1 — apenas ativar/desativar):**
*   Toggle de ativo/inativo com modal de confirmação
*   Ao desativar: "Tipos inativos não serão usados em novas detecções. Anomalias já registradas com este tipo são mantidas."
**Fase 2 (não implementar agora):**
*   Botão "Novo Tipo" com modal de criação
*   Campos: Nome · Código (único, sem espaços) · Severidade · Descrição
